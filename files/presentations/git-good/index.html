Why source control?

live_help "Uh, what is source control?"

Also known as version control, source control really just means keeping structured backups of your work.

  • Dropbox
  • Time Machine or manual backups
  • Wikis

But there’s a lot more to it than just keeping backups, as we’ll get to soon. First - why do you need it?

Get going with

We’re going to focus on command line git today, but there are a number of graphical tools available too.

How does work?


  • Git maintains a hidden folder in your project

  • You commit snapshots of changes to this folder

  • Git keeps track of everything

live_help "Why is this better than just backing up?"

It’s better because of MANY REASONS:

  • It’s designed for text: you can see exactly what changed
  • It’s designed for collaboration and will let you share code and documents more easily
  • It’s distributed: there’s no one “central” version
  • You get the provenance of each change
  • You have every snapshot, which is hugely useful for figuring out when something broke
  • And you can use it as a backup system anyway

Setting up

  1. Linux: get git via your package manager; for example, sudo apt-get install git

  2. OSX: may install automatically if you run git in the terminal, or go to git-scm.org/download/mac

  3. Windows: get a better console from cmder.net or try git-for-windows.github.io

Then identify yourself:

$ git config --global user.name "Scott Thomas"
$ git config --global user.email "swt30@ast.cam.ac.uk"

Set up a repository

$ git init

Initialized empty Git repository in /home/sthomas/Documents/my-project/.git/

Make an empty directory and do git init.

A repository tracks the files in your project and any changes to them. It doesn’t matter what’s inside it - just know that this folder holds the key to your dreams.

Show some commitment

Open up your favourite text editor and save a few lines of text into a README document (say, “README.md”). Then:

$ git add README.md

$ git commit

[master (root-commit) 52b46ae] Add README
 1 file changed, 2 insertions(+)
 create mode 100644 README.md

you may need to tell git which editor to use:

$ git config --global core.editor= ...

What just happened?

$ git log

commit a04cbdc1ab5943837eb2c1a6f80c3e725a45be6d
Author: Scott Thomas <swt30@ast.cam.ac.uk>
Date:   Mon Jun 6 17:51:21 2016 +0100

$ git status

On branch master
nothing to commit, working directory clean

The log shows the commit

  • it gets a specific identifier called a hash
  • it has all the details
  • if you ever publish the repository, this is all public info

The status shows the current state of the directory

  • at the moment it’s clean - git knows of no more changes

The two-step commit process

  • 0. Edit
    Make changes as normal…
  • 1. Stage
    Mark any changes that you want to include…
  • 2. Commit
    Lock it in!

git status will show you what’s been added and what’s waiting.


  1. computer Change or add to the text in README, git add it, then change the text again and check git status. What happened? Commit those changes too.

We had to stage (add) the changes and then commit everything.

This process means that:

  1. You can commit just a bit at a time, even if you’ve changed lots of things
  2. You can review what you’ve done before you commit (handy)

After the exercise, it should show as modified again. If you committed now, your most recent changes won’t be added. The best way to think about this is that git add adds changes, not files.

The three key commands

  1. git status: Check which files have been changed

  2. git add <files or folders>: Stage the changes

  3. git commit: Lock it in

Git for you

So in this section, “Git for You”, we’ll look at the most useful features that Git provides the lone astronomer.

Useful tools for the lone astronomer

  • git checkout
    lets you move between commits
  • git tag
    marks a commit for later reference
  • git reset
    is your “undo” button

Specifically, these features are: tagging interesting points; reverting to earlier versions of your code; and backing up online.

Use git checkout to move between commits

$ git log

commit ffa75e3a59654373383ec6554de8a8d58df29092
Author: Scott Thomas <swt30@ast.cam.ac.uk>
Date:   Mon Jun 6 20:17:35 2016 +0100

    Add author to README

commit 52b46aeec2fb86d32e6d8925358997e289140bd8
Author: Scott Thomas <swt30@ast.cam.ac.uk>
Date:   Mon Jun 6 18:30:41 2016 +0100

    Add README

 README.md | 2 ++
 1 file changed, 2 insertions(+)

Here’s an example. I use git log to get the ID of the commit I need… I copy it [the lower one]… And I move back to that commit.

$ git checkout 52b46aee

  Note: checking out '52b46aee'.

You are in 'detached HEAD' state. You can look around, make experimental
changes and commit them, and you can discard any commits you make in this
state without impacting any branches by performing another checkout.

If you want to create a new branch to retain commits you create, you may
do so (now or later) by using -b with the checkout command again. Example:

  git checkout -b <new-branch-name>

HEAD is now at 52b46ae... Add README
  1. computer git checkout your first commit and look at the README. Use git checkout master to return to where you started.

What’s the use scenario here?

  • See if something worked with an earlier version of your analysis code
  • You made a stupid commit and you want to go back
  • git checkout also lets you reset files to an earlier version

It’s impossible to lose your work when you do a checkout and go back to an old version. Git will warn you if you need to commit files first.

Tagging interesting commits

$ git tag original-plots-for-paper
$ git checkout original-plots-for-paper

Use git tag to give a label to a particular commit and more easily revisit it later. (I use this to tag submitted versions of plots or papers, for example).

Now I can refer to that commit using a more intuitive name. Tags are useful but, if you’re going to forget one thing you learned today, forget this, because they’re entirely optional.

Undoing your screw-ups

  1. git reset -- <file> unstages a file once you’ve git added it

  2. git reset --hard discards all unstaged changes and reverts to the last commit.

  3. git checkout <file> is another way of achieving the same thing.

(DANGER)

  1. computer Make a new file and stage it. Decide you don’t actually want it after it. Use git reset to unstage it. Delete it.

  2. computer Delete your README. Then bring it back from the dead by restoring to the last commit.

The reason I say “danger!” is because this command will discard any unstaged changes. Remember kids, if you want to keep it, commit it!

There’s usually a million different ways to do things in git.

These three commands will let you move back and forward through time itself - pretty cool, huh? In the final section, we’ll look at working online and collaborating with others.

Your local repository doesn’t know about the one on github yet. Adding a reference to the remote repository is easy, and will let you push and pull changes to keep your Github copy in sync.

Send your changes

  1. first time: git push -u origin master
  2. subsequently: git push
$ git push -u origin master

  1. computer Do this and take a look at your shiny new github repository!

This is called “pushing”. When you push, you send your latest commits to the remote repository.

Get a copy of something

git clone https://github.com/username/repo.git

  1. computer Delete your local repository and then clone a fresh copy.
  2. computer Download, or fork and download my repository: https://github.com/swt30/git-good.git

This is called cloning. Cloning gives you an exact duplicate of the repository, including its entire history.

You can do this with:

  • a repository that someone else has made that you want a copy of
  • your own repository when you want to work on it from another machine
  • a fork of someone else’s repository when you want to be able to modify it and submit your changes

Update a local copy

git pull

  1. computer I’m going to make a change to the repository you just cloned. Once I do, use git pull to get it.

This “pulls” in changes from elsewhere. For this example, I’m going to push some changes from my machine to the repository you just cloned from github. Then you are going to pull in the changes.
